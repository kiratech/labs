# Lab | Manage Ansible Operator packages

In this lab you will define a method to package and version the Ansible based
Kubernetes Operator.

## Make the original working directory a Git repository

repository. This can be done by using `git init`:


## Generate the first YAML package

It is time to generate the first YAML package for version `0.0.1`, this
procedure is based on the `kustomize` command that was installed during the
previous lab, [Kubernetes-Operators-SDK-Create.md](Kubernetes-Operators-SDK-Create.md).

If `make deploy` was not launched, it is possible to install the `kustomize`
command by using this:

```console
$ make kustomize
(no output)
```

The `kustomize` command is crucial to generate the YAML and must be invoked with
the `build` argument referring to the `confg/default` subdirectory, which points
to the structure that was generated by the SDK:

```console
$ bin/kustomize build config/default --output v0.0.1.yaml
(no output)
```

The resulting `v0.0.1.yaml` file will contain the entire operator's declaration,
it will be the "package" of what have been developed.

Now everything can be committed:

```console
$ git add . && git commit -m "Version v0.0.1"
[main 1649dc2] Version v0.0.1
 1 file changed, 391 insertions(+)
 create mode 100644 v0.0.1.yaml
```

If the `make deploy` was previously launched, applying this newly generated YAML
should produce no change in the Kubernetes cluster:

```console
$ kubectl apply -f v0.0.1.yaml
namespace/kiraop-system unchanged
customresourcedefinition.apiextensions.k8s.io/akits.kiratech.io unchanged
serviceaccount/kiraop-controller-manager unchanged
role.rbac.authorization.k8s.io/kiraop-leader-election-role unchanged
```

## Generate a new version

To simulate a new version generation, let's change something in our operator, by
adding a custom welcome page to the web server it deploys.

Change `roles/akit/tasks/main.yml` by adding:

```yaml
- name: Create Akit ConfigMap.
  kubernetes.core.k8s:
    state: present
    resource_definition:
      apiVersion: v1
      kind: ConfigMap
      metadata:
        name: index-html
        namespace: "{{ akit_namespace }}"
      data:
        index.html: |
          This is my faboulous Webserver inside {{ akit_namespace }}!
```

And change the deployment to mount the configmap:

```yaml
- name: Create Akit Deployment.
  kubernetes.core.k8s:
    state: present
    resource_definition:
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: akit
        namespace: "{{ akit_namespace }}"
        labels:
          app: akit
      spec:
        replicas: 1
        selector:
          matchLabels:
            app: akit
        template:
          metadata:
            labels:
              app: akit
          spec:
            containers:
              - name: akit
                image: "{{ akit_container }}"
                volumeMounts:
                - name: docroot
                  mountPath: /usr/share/nginx/html
            volumes:
              - name: docroot
                configMap:
                  name: index-html
```

Then, build and push the new `v0.0.2` tag:

```console
$ IMG=quay.io/mmul/kiraop:v0.0.2 make docker-build docker-push
docker build -t quay.io/mmul/kiraop:v0.0.2 .
[+] Building 0.8s (11/11) FINISHED
...
v0.0.2: digest: sha256:3668ebcf8052b31ea4dd1d6aaa8130168a0b0b1fd791ee4de6665baf96b104ec size: 3452
```

The new YAML package will still be generated by using `kustomize`, but first the
`/config/manager/kustomization.yaml` should be accordingly updated.

There are two possible ways to do this:

- Use the `IMG=quay.io/mmul/kiraop:v0.0.2 make deploy` command, that will
  also install on the destination Kubernetes cluster the new version.
- Manually generate the new files, by using:

  ```console
  $ (cd config/manager && ../../bin/kustomize edit set image controller=quay.io/mmul/kiraop:v0.0.2)
  (no output)
  ```

In both cases the result will be this variation on the
`config/manager/kustomization.yaml` file:

```console
$ git diff config/manager/kustomization.yaml
diff --git a/config/manager/kustomization.yaml b/config/manager/kustomization.yaml
index 7f645a2..f15f6af 100644
--- a/config/manager/kustomization.yaml
+++ b/config/manager/kustomization.yaml
@@ -5,4 +5,4 @@ kind: Kustomization
 images:
 - name: controller
   newName: quay.io/mmul/kiraop
-  newTag: v0.0.1
+  newTag: v0.0.2
```

Finally, to create the all-in-one release file, named `v0.0.2.yaml`, use
`kustomize`:

```console
$ bin/kustomize build config/default --output v0.0.2.yaml
(no output)
```

And commit everything as before:

```console
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   config/manager/kustomization.yaml
        modified:   roles/akit/tasks/main.yml

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        v0.0.2.yaml

no changes added to commit (use "git add" and/or "git commit -a")

$ git add . && git commit -m "Version v0.0.2"
[main 15f41e8] Version v0.0.2
 3 files changed, 412 insertions(+), 1 deletion(-)
 create mode 100644 v0.0.2.yaml
```

Now everything is tracked.

## Deploying specific releases

To apply the new release into the cluster, the `v0.0.2.yaml` can be passed as is
to `kubectl`, that will configure just the changed parts (usually the controller
manager deployment):

```console
$ kubectl apply -f v0.0.2.yaml
...
clusterrolebinding.rbac.authorization.k8s.io/kiraop-proxy-rolebinding unchanged
service/kiraop-controller-manager-metrics-service unchanged
deployment.apps/kiraop-controller-manager configured
...
```

The new deployment will be started and once it will come up the old one will be
terminated:


## Using Bundles

Another way to distribute an operator is by using Bundles, which are OLM
(_Operator Lifecycle Manager_) compatible packaging units containing the
operator manifests, its metadata, and the _ClusterServiceVersion_ (CSV) that
OLM needs to manage the operator lifecycle.

The OLM topic is explored in the specific
[Kubernetes-Operators-OLM-Operators.md](Kubernetes-Operators-OLM-Operators.md)
lab, but for completeness, this is the procedure to generate a bundle:

```console
$ IMG=quay.io/mmul/kiraop:v0.0.1 make bundle
/usr/local/bin/operator-sdk generate kustomize manifests -q

Display name for the operator (required):
> Kiraop

Description for the operator (required):
> Kiratech Operators Training Operator

Provider's name for the operator (required):
> Kiratech S.p.A.

Any relevant URL for the provider name (optional):
> https://www.kiratech.it

Comma-separated list of keywords for your operator (required):
> sample,training

Comma-separated list of maintainers and their emails (e.g. 'name1:email1, name2:email2') (required):
> kirater:kirater@kiratech.it

cd config/manager && /home/kirater/operators/kiraop/bin/kustomize edit set image controller=quay.io/mmul/kiraop:v0.0.1
/home/kirater/operators/kiraop/bin/kustomize build config/manifests | /usr/local/bin/operator-sdk generate bundle -q --overwrite --version 0.0.1
WARN[0000] ClusterServiceVersion validation: [CSVFileNotValid] (kiraop.v0.0.1) csv.Spec.minKubeVersion is not informed. It is recommended you provide this information. Otherwise, it would mean that your operator project can be distributed and installed in any cluster version available, which is not necessarily the case for all projects.
INFO[0000] Creating bundle.Dockerfile
INFO[0000] Creating bundle/metadata/annotations.yaml
INFO[0000] Bundle metadata generated successfully
/usr/local/bin/operator-sdk bundle validate ./bundle
WARN[0000] Warning: Value : (kiraop.v0.0.1) csv.Spec.minKubeVersion is not informed. It is recommended you provide this information. Otherwise, it would mean that your operator project can be distributed and installed in any cluster version available, which is not necessarily the case for all projects.
INFO[0000] All validation tests have completed successfully
```

The metadata will be asked the first time the command is launched for a given
operator, and will be stored in
`config/manifests/bases/<operator>.clusterserviceversion.yaml` for subsequent
runs. This information is used by operator marketplaces (such as OperatorHub
or the OpenShift one) to describe and categorise the operator.

The two `WARN` lines about `minKubeVersion` are non-critical and can be
ignored for development purposes. To suppress them, set the minimum Kubernetes
version in the generated CSV manifest.

Once the bundle directory is generated, there are two additional targets to
build and publish it:

```console
$ IMG=quay.io/mmul/kiraop:v0.0.1 make bundle-build bundle-push
...
```

These commands build a **bundle container image** (a standard OCI image that
wraps the contents of the `bundle/` directory) and push it to the remote
registry. OLM does not consume the raw `bundle/` directory directly; it pulls
and unpacks this container image to retrieve the operator manifests and
metadata needed to automate the installation.
